# 智能站点接入工作流设计（V1）

## 1. 文档目标

本文档定义 NetWorkspace 中“用户只提供链接，系统自动完成站点接入”的固定工作流。

目标不是做一个完全自由发挥的 Agent，而是建立一套：

- 用户输入最小化
- 系统流程固定化
- AI 辅助增强
- 全链路可审计

的站点接入机制。

## 2. 核心目标

用户理想体验应为：

1. 粘贴一个网站链接
2. 系统识别站点类型
3. 系统生成模板草稿
4. 系统自动试抓几条数据
5. 用户确认是否正确
6. 模板进入个人工作区并可立即使用

## 3. 设计原则

### 3.1 用户输入尽量少

默认只需要：

- 一个目标链接

可选补充：

- 这是什么类型的网站
- 想抓的对象是职位、公司还是内容
- 是否需要详情页

### 3.2 工作流必须固定

不能让 Agent 每次自由决定怎么做。  
应通过固定节点执行，保证可解释性和稳定性。

### 3.3 AI 是增强层，不是唯一决策者

AI 可以：

- 看懂页面
- 生成模板草稿
- 提供修复建议

但最终发布必须经过：

- 系统验证
- 用户确认
- 必要时管理员审核

### 3.4 网络补充信息只能作为辅助依据

公开搜索结果、站点文档、社区讨论等可以帮助理解站点，但不能代替对真实页面的验证。

## 4. 工作流总览

完整工作流建议分为 9 个阶段：

1. 用户提交链接
2. 系统做基础识别
3. 系统抓取目标页面
4. AI / 多模态理解页面结构
5. 网络补充信息检索
6. 生成模板草稿
7. 小样本测试与质量评估
8. 用户确认与发布
9. 后续监控与修复

## 5. 阶段详细说明

### 5.1 阶段一：用户提交链接

输入：

- `target_url`

可选输入：

- `content_type_hint`
- `needs_detail_page`
- `workspace_id`

要求：

- 支持首页、搜索页、列表页、详情页作为入口
- 对明显无效 URL 直接拦截

输出：

- `onboarding_request`

### 5.2 阶段二：系统做基础识别

系统自动识别：

- 域名
- 页面类型
- 是否疑似招聘站
- 页面渲染类型
- 是否疑似需要登录
- 是否疑似存在反爬

建议输出对象：

```json
{
  "domain": "example.com",
  "page_type": "search_list",
  "site_guess": "job_site",
  "render_mode": "browser",
  "risk_flags": ["dynamic_render"]
}
```

### 5.3 阶段三：系统抓取目标页面

目的：

- 获取真实页面内容，作为 AI 和规则生成的主依据

建议抓取内容：

- HTML
- 截图
- 网络请求摘要
- DOM 结构摘要

说明：

- 若为浏览器渲染型页面，应优先保存渲染后 DOM 与截图

### 5.4 阶段四：AI / 多模态理解页面结构

AI 在这一阶段应完成：

1. 识别页面主要区域
2. 判断是否存在重复职位卡片
3. 识别可能的字段位置
4. 判断是否存在分页入口
5. 判断是否需要详情页

推荐输出：

- 列表容器候选
- item selector 候选
- 字段 selector 候选
- 分页规则候选
- 置信度

### 5.5 阶段五：网络补充信息检索

目的：

- 为复杂站点提供额外线索

可检索信息：

- 站点公开介绍
- 帮助文档
- robots / sitemap
- 搜索页参数线索
- 社区中公开的页面结构讨论

边界：

- 只能作为辅助证据
- 最终模板必须回到真实页面验证

### 5.6 阶段六：生成模板草稿

系统基于真实页面 + AI 分析结果 + 网络补充信息，生成 DSL 草稿。

输出结果应包括：

- 顶层模板结构
- 列表规则
- 详情规则
- 分页规则
- 字段映射
- 标准化建议
- 去重建议

默认状态：

- `draft`

### 5.7 阶段七：小样本测试与质量评估

系统自动试抓：

- 1 页列表
- 3 到 5 条详情

评估指标：

- 记录识别数
- 标题字段完整率
- 链接字段完整率
- 详情页成功率
- 去重后有效记录数
- 规则置信度

质量输出建议：

```json
{
  "sample_size": 5,
  "list_items_found": 20,
  "detail_success_rate": 0.8,
  "field_completeness": {
    "job_title": 1.0,
    "company_name": 0.95,
    "salary_text": 0.75
  },
  "confidence": "medium"
}
```

### 5.8 阶段八：用户确认与发布

前端需要向用户展示：

- 识别出的站点类型
- 模板抓取到的样本结果
- 字段完整率
- 哪些字段高置信，哪些字段低置信
- 是否发现分页
- 是否建议补抓详情页

用户操作选项：

- 保存为个人模板
- 重新生成
- 手动编辑
- 放弃

发布后模板状态建议进入：

- `validated`
- 经用户确认后进入 `active`

### 5.9 阶段九：后续监控与修复

模板一旦投入使用，系统应持续监控：

- 采集成功率
- 字段完整率变化
- 分页失效
- 详情页失效
- 页面结构漂移

触发修复时机：

- 成功率明显下降
- 关键字段缺失率升高
- 连续多个任务失败

此时再进入“模板修复工作流”。

## 6. 模板修复工作流

当模板失效时：

1. 系统抓取当前页面
2. 对比旧模板与当前页面结构
3. AI 提供修复建议
4. 小样本验证修复效果
5. 用户确认后生成新版本

输出：

- 新旧模板 diff
- 修复前后测试结果对比

## 7. 人机分工

### 7.1 系统负责

- 链接校验
- 页面抓取
- 站点基础识别
- 测试执行
- 质量评估
- 状态流转
- 审计记录

### 7.2 AI 负责

- 页面结构理解
- 字段候选识别
- 模板草稿生成
- 修复建议
- 解释与摘要

### 7.3 用户负责

- 提供目标链接
- 确认模板结果是否符合预期
- 决定是否发布到个人工作区

### 7.4 管理员负责

- 审核共享提案
- 关注模板健康度
- 挑选具有普适价值的模板进入官方能力库

## 8. 状态机建议

站点接入请求状态建议：

- `submitted`
- `identified`
- `analyzing`
- `draft_generated`
- `testing`
- `awaiting_confirmation`
- `published`
- `failed`
- `abandoned`

## 9. 审计与透明度要求

每次接入都应记录：

- 用户提交的链接
- 系统抓取的页面快照
- AI 输出的模板草稿
- 网络补充信息来源
- 测试结果
- 用户最终操作

前端应能展示：

- “为什么识别成这个字段”
- “依据了哪些页面或截图”
- “AI 的置信度如何”
- “这次模板和旧版本差在哪里”

## 10. 风险与处理

### 10.1 页面结构复杂

处理：

- 降低自动发布概率
- 强制用户确认
- 标记低置信度

### 10.2 反爬或登录限制

处理：

- 标记风险
- 不立即承诺可接入
- 保留失败诊断结果

### 10.3 AI 幻觉

处理：

- 所有结果必须经过小样本验证
- 不允许 AI 草稿直接成为生产模板

### 10.4 网络补充信息误导

处理：

- 公开检索结果只作辅助依据
- 真正执行规则必须基于当前真实页面

## 11. 对应前端页面建议

建议新增“智能站点接入”页面，分为四步：

1. 输入链接
2. 系统分析
3. 结果预览
4. 确认发布

关键展示组件：

- 站点画像卡片
- 页面截图预览
- 模板字段预览表
- 样本结果表格
- 置信度标签
- 错误诊断面板

## 12. 对应后端接口建议

建议至少包含以下接口：

- `POST /onboarding/requests`
- `GET /onboarding/requests/{id}`
- `POST /onboarding/requests/{id}/analyze`
- `POST /onboarding/requests/{id}/generate-template`
- `POST /onboarding/templates/{id}/test`
- `POST /onboarding/templates/{id}/publish`

## 13. 结论

NetWorkspace 的“智能站点接入”不应被设计成自由式 Agent，而应被设计成：

- 以用户链接为起点
- 以固定工作流为骨架
- 以 AI 和多模态为增强能力
- 以验证、确认、审计为安全边界

的标准化接入流程。
