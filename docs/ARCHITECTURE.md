# 系统架构设计（V1）

## 1. 设计目标

NetWorkspace 的架构目标不是单纯“把爬虫跑起来”，而是支撑以下能力：

1. 多网站接入
2. 异步采集与历史沉淀
3. 个人工作区隔离
4. AI 辅助模板生成与修复
5. 搜索、分析、导出和可视化
6. 管理员审核与平台演化

## 2. 总体分层

系统建议分为 8 层：

1. 前端工作台层
2. API 服务层
3. 业务服务层
4. 任务调度层
5. 采集执行层
6. AI / Agent 层
7. 数据存储层
8. 知识库与审计层

## 3. 各层职责

### 3.1 前端工作台层

职责：

- 登录与权限入口
- 仪表板、任务中心、模板中心、分析中心
- 个人工作区配置
- Agent 对话界面
- 采集预览与模板确认

推荐技术：

- React
- TypeScript
- Vite
- Ant Design
- ECharts

### 3.2 API 服务层

职责：

- 提供 REST API
- 鉴权与会话校验
- 参数校验
- 返回统一 JSON 响应

推荐技术：

- FastAPI
- Pydantic

### 3.3 业务服务层

职责：

- 用户与工作区管理
- 模板管理
- 任务创建与状态流转
- 搜索、导出、分析逻辑
- 提案与审核逻辑

建议：

- 路由层只做入参处理和响应封装
- 核心逻辑放到 services / repositories

### 3.4 任务调度层

职责：

- 异步执行采集任务
- 重试与失败处理
- 定时调度
- 任务日志记录

推荐技术：

- Redis
- Celery 或 RQ

### 3.5 采集执行层

职责：

- 调用不同类型的采集器
- 执行官方模板和个人模板
- 获取列表页和详情页数据
- 将结果交给标准化流水线

采集器建议分为：

- `ApiSpider`
- `HtmlSpider`
- `BrowserSpider`

### 3.6 AI / Agent 层

职责：

- 站点识别
- 模板草稿生成
- 模板修复建议
- 语义字段抽取
- 分析报告生成
- 工作区辅助搭建

原则：

- AI 负责辅助，不直接成为唯一真值来源
- 所有关键输出必须可预览、可审计、可确认

### 3.7 数据存储层

职责：

- 存储用户、工作区、模板、任务、结构化数据、原始数据
- 支撑搜索与分析
- 保留历史快照

推荐技术：

- PostgreSQL 优先
- MySQL 8 可兼容

### 3.8 知识库与审计层

职责：

- 平台知识库
- 工作区知识库
- Agent 操作依据记录
- 模板和配置变更 diff
- 管理员审计日志

## 4. 关键对象

系统应围绕以下核心对象设计：

- `User`
- `Workspace`
- `Template`
- `TemplateVersion`
- `CrawlTask`
- `TaskRunLog`
- `JobRecord`
- `Source`
- `Proposal`
- `AuditLog`
- `KnowledgeDocument`

## 5. 典型数据流

### 5.1 智能站点接入

1. 用户提交链接
2. 系统识别站点类型
3. Agent 生成模板草稿
4. 系统小样本验证
5. 用户确认
6. 模板保存到个人工作区

### 5.2 采集任务执行

1. 用户创建任务
2. API 写入任务表
3. Worker 拉取任务
4. 采集器执行模板
5. 数据进入标准化流水线
6. 原始数据和结构化数据落库
7. 更新任务状态和日志

### 5.3 分析与问答

1. 用户发起筛选或自然语言问题
2. 系统解析意图
3. 查询结构化数据和历史快照
4. 生成图表与摘要
5. 返回前端展示

## 6. 设计边界

V1 明确不做：

- 允许用户执行任意 Python 采集脚本
- Agent 直接修改主系统生产代码
- 未经确认的模板自动进入生产可用状态

## 7. 结论

NetWorkspace 的本质是“稳定主系统 + 可演化个人工作区 + AI 辅助能力层”，而不是把所有能力都硬编码在一个网站里。
